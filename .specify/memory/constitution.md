# Polls Constitution

## Core Principles

### I. Type Safety End-to-End (NON-NEGOTIABLE)

TypeScript is the only implementation language for both `backend/` (NestJS 10) and
`frontend/` (React 19 + Vite). `npm run check:ts` MUST pass at the workspace root
before any merge — both packages run `tsc --noEmit` in CI. Use of `any`, `as unknown as`,
or `// @ts-ignore` requires an inline justification comment naming the constraint that
forces it. Runtime input at every external boundary MUST be validated: incoming HTTP
payloads via `class-validator` DTOs (backend), incoming form/API data via Zod schemas
(frontend). Internal function boundaries trust their types; do not double-validate.

**Rationale**: The product surface area is small but spans two TS toolchains, a Prisma
schema, and a REST contract. Compile-time guarantees plus boundary validation are what
keep refactors safe without paying for them in test mocks.

### II. Test Discipline & Green CI

Every feature MUST land with the appropriate level of automated coverage and CI MUST be
green before merge. The minimum bar:

- Backend services and pure utilities: Jest unit tests co-located as `*.spec.ts` next
  to the implementation.
- Backend HTTP behavior that crosses module boundaries (auth, response submission,
  ownership checks, rate limits): supertest e2e tests under `backend/test/`.
- Frontend pure logic, hooks, and components with non-trivial state: Vitest tests under
  `__tests__/` adjacent to the source.
- Frontend critical user journeys (admin flows, poll submission): Playwright specs
  under `frontend/e2e/`.

CI on every push and pull request runs: `check:ts`, unit tests with coverage, and e2e
tests for the changed package. A failing test or a TypeScript error blocks merge — no
`--no-verify`, no skipped suites, no `it.skip` left in `main`.

**Rationale**: The backend owns auth, ownership, and rate-limiting logic where regressions
are silent and security-relevant; the frontend owns admin workflows that aren't unit-testable
in isolation. The two test layers per package exist precisely because mocks alone hide
integration bugs.

### III. Schema-First Data Modeling

The PostgreSQL schema is owned by `backend/prisma/schema.prisma`. All schema changes
MUST go through `prisma migrate` and land migration files under `backend/prisma/migrations/`.
Direct DDL against any environment is forbidden. Naming conventions are enforced by the
existing schema and MUST be followed: camelCase fields in Prisma models, snake_case
column/table names mapped via `@map`/`@@map`, UUID primary keys generated by Prisma,
foreign-key cascade behavior chosen explicitly per relation. The Prisma client is
regenerated (`prisma generate`) as part of the build; never check generated client code
into source.

**Rationale**: Prisma is the single source of truth for the data layer. Drift between
the live schema and `schema.prisma` would silently break type-safe queries throughout
the backend, which is the entire reason this stack was chosen.

### IV. Versioned REST API with Documented Contract

The HTTP API is exposed under `/api/v1` and documented via `@nestjs/swagger`. Endpoint
contracts (path, method, request DTO, response shape, status codes) are part of the
public surface and MUST NOT change in backward-incompatible ways within `v1`. Permitted
within `v1`: additive fields, new endpoints, new optional query parameters. Forbidden
within `v1`: removing or renaming fields/endpoints, tightening validation on existing
endpoints, changing status code semantics. A breaking change requires a new
`/api/v{N+1}` namespace and a documented deprecation window for `vN`.

**Rationale**: The frontend, the public poll links, and any future integrations all
consume `/api/v1`. Stability of this surface is the contract that lets the two
workspaces evolve independently.

### V. Security by Default

Authentication uses the JWT access + refresh token pattern with the following
non-negotiables: passwords hashed with `bcrypt` (never stored or logged in plaintext);
refresh tokens stored as hashes only (`refresh_tokens.token_hash`) and rotated on use;
role-based authorization (`USER` / `ADMIN`) enforced server-side via guards on every
protected route — never relied upon client-side; rate limiting via `@nestjs/throttler`
on authentication endpoints and submission endpoints; secrets (`JWT_*`, `DATABASE_URL`)
sourced from environment only and never committed. Poll access tokens (the public `slug`
or `accessToken` link) MUST be treated as bearer credentials for the unauthenticated
respondent flow — server-side checks for poll `isActive` and `expiresAt` apply on every
submission.

**Rationale**: This is a polling/survey product where one ADMIN bypass leaks every
respondent's answers. The security model has to be simple enough to verify by reading
the guards, which means the rules above are mechanical and uniform rather than
case-by-case.

## Technology Stack & Constraints

The stack is fixed and changes require a constitution amendment:

- **Runtime**: Node.js `>=20.0.0`, npm `>=10.0.0` (enforced via `package.json` `engines`).
- **Monorepo**: npm workspaces with `backend` and `frontend` packages; root scripts
  (`build`, `test`, `check:ts`, `lint`) fan out via `--workspaces --if-present`.
- **Backend**: NestJS 10, Prisma 5, PostgreSQL 16, `passport-jwt`, `class-validator`,
  `@nestjs/swagger`, `@nestjs/throttler`, Jest + supertest.
- **Frontend**: React 19, Vite 5, React Router 7, TanStack Query 5, Zustand, Zod,
  react-hook-form, Tailwind 3, Recharts, Vitest + Testing Library, Playwright.
- **Containerization**: Each workspace ships a `Dockerfile`; the full stack is composed
  via the root `docker-compose.yml` (postgres + backend + frontend). Production
  deploys MUST run `prisma migrate deploy` before starting the backend (already wired
  in `docker-compose.yml`).
- **Configuration**: All runtime configuration via environment variables, loaded via
  `@nestjs/config` (backend) and Vite's `VITE_` env mechanism (frontend). No hardcoded
  hosts, ports, or secrets in source.

Adding a new top-level dependency that overlaps with an existing one (e.g., a second
state manager, a second HTTP client, a second form library) is forbidden without a
documented migration plan that retires the old one.

## Development Workflow & Quality Gates

- **Branching**: Work happens on feature branches; `main` is protected. PRs target
  `main`.
- **Commit style**: Conventional prefixes are used in history (`feat:`, `fix:`, `docs:`,
  `refactor:`, `chore:`). Continue this style; the prefix is the contract.
- **Pre-merge gates** (all enforced by GitHub Actions on every push and PR):
  1. `check:ts` passes for the changed workspace(s).
  2. Unit tests pass with coverage uploaded as an artifact.
  3. E2E tests pass (backend: supertest against a real Postgres service container;
     frontend: Playwright against a real backend instance).
  4. Lint passes (`npm run lint`).
- **Local dev**: `docker compose up` brings up the full stack; `npm run start:dev` runs
  the backend in watch mode against an externally-running Postgres.
- **Migrations in PRs**: Any PR that modifies `schema.prisma` MUST include the generated
  migration file in `backend/prisma/migrations/` in the same commit.
- **Code review**: At least one reviewer MUST verify the PR against this constitution.
  Reviewers explicitly check the four gates above plus principle compliance (type
  safety, test coverage at the required layer, schema-first changes, API
  backward-compatibility, security-by-default).

## Governance

This constitution supersedes informal conventions and ad-hoc decisions. When this
document and another guidance file disagree, this document wins; the other file MUST
be updated.

**Amendments**: Any change to this constitution is made via a PR that modifies
`.specify/memory/constitution.md`, bumps the version per the policy below, updates the
`Last Amended` date to the merge date in ISO `YYYY-MM-DD` format, and includes a Sync
Impact Report comment at the top of the file describing the delta and any dependent
templates touched.

**Versioning policy** (semantic):

- **MAJOR**: A principle is removed, redefined in a backward-incompatible way, or
  governance rules change such that prior PRs would no longer comply.
- **MINOR**: A new principle or section is added, or existing guidance is materially
  expanded (new mandatory rule, new gate).
- **PATCH**: Wording clarifications, typo fixes, non-semantic refinements that do not
  change what is required of contributors.

**Compliance review**: Every PR review is the compliance checkpoint. Violations that
must be merged anyway (rare) are tracked in the PR description with explicit
justification, and create a follow-up issue to either remediate or amend this
constitution. Repeated identical justifications are a signal that the principle needs
amendment, not that exceptions should keep accruing.

**Runtime guidance**: Day-to-day implementation details (commands, file layouts,
in-flight plans) live in `CLAUDE.md` and the active feature plan under `specs/`. This
constitution defines the rules; those files describe the current work.
